# 模块十八：锁与权限

## 学习目标

- 理解全局锁、表级锁、行级锁的区别

- 理解共享锁和排他锁的区别

- 理解死锁的原理和处理方法

## 1. 锁的级别

### 🏠 生活类比

> **全局锁** = 整栋楼封锁
> **表级锁** = 封锁一层楼
> **行级锁** = 只锁一间房

### 📖 概念讲解

| 锁级别 | 说明 | 支持引擎 |
|--------|------|---------|
| 全局锁 | 锁整个数据库实例 | FLUSH TABLES WITH READ LOCK |
| 表级锁 | 锁整张表 | MyISAM / InnoDB |
| 行级锁 | 锁表中特定行 | InnoDB |

### 💻 SQL 代码示例

```sql
CREATE DATABASE lock_demo;
USE lock_demo;

CREATE TABLE t_account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10,2)
) ENGINE = InnoDB;
INSERT INTO t_account VALUES (1, 'Alice', 10000), (2, 'Bob', 5000);

-- 表级锁（InnoDB）
LOCK TABLE t_account READ;  -- 读锁，其他会话只能读
-- 其他会话：SELECT * FROM t_account; -- 可读
-- 其他会话：INSERT INTO t_account VALUES (3, 'Carol', 3000); -- 等待

UNLOCK TABLES;

LOCK TABLE t_account WRITE;  -- 写锁，独占
-- 其他会话无法读写

UNLOCK TABLES;
```

### ❌ 常见错误

- ❌ 全局锁不当使用 → 所有写操作被阻塞

## 2. InnoDB 行级锁类型

### 📖 概念讲解

| 锁类型 | 说明 |
|--------|------|
| 记录锁（Record Lock） | 锁定索引记录 |
| 间隙锁（Gap Lock） | 锁定索引之间的间隙 |
| 临键锁（Next-Key Lock） | 记录锁 + 间隙锁组合 |
| 插入意向锁 | 插入操作前在间隙加的锁 |

**InnoDB 通过临键锁防止幻读**（REPEATABLE READ 隔离级别下）

### 💻 SQL 代码示例

```sql
USE lock_demo;

CREATE TABLE t_order (
    id INT PRIMARY KEY,
    status VARCHAR(20),
    INDEX idx_status (status)
);
INSERT INTO t_order VALUES
    (1, 'paid'), (3, 'paid'), (5, 'shipped'), (7, 'delivered');

-- 记录锁：锁定特定记录
-- SELECT * FROM t_order WHERE id = 3 FOR UPDATE;
-- 锁定 id=3 的索引记录

-- 间隙锁：锁定 id 在 1 和 3 之间的间隙
-- SELECT * FROM t_order WHERE id > 1 AND id < 5 FOR UPDATE;
-- 锁定 id IN (2, 4) 的间隙

-- 临键锁：记录锁 + 间隙锁
-- SELECT * FROM t_order WHERE id >= 1 AND id <= 5 FOR UPDATE;
```

### ❌ 常见错误

- ❌ 以为行锁锁的是整行数据 → 行锁锁的是索引项

## 3. 共享锁和排他锁

### 📖 概念讲解

| 锁类型 | 说明 | 兼容情况 |
|--------|------|---------|
| 共享锁（S锁） | 读锁，多事务可同时持有 | S 与 S 兼容，X 与任何锁不兼容 |
| 排他锁（X锁） | 写锁，独占 | X 与任何锁都不兼容 |

### 💻 SQL 代码示例

```sql
USE lock_demo;

-- 共享锁（S锁）
-- Session 1:
START TRANSACTION;
SELECT * FROM t_account WHERE id = 1 LOCK IN SHARE MODE;
-- Session 2:
START TRANSACTION;
SELECT * FROM t_account WHERE id = 1 LOCK IN SHARE MODE;  -- ✅ 可获取，多个S锁共存

-- 排他锁（X锁）
-- Session 1:
START TRANSACTION;
SELECT * FROM t_account WHERE id = 1 FOR UPDATE;  -- 获取X锁
-- Session 2:
START TRANSACTION;
SELECT * FROM t_account WHERE id = 1 FOR UPDATE;  -- 等待，X锁冲突
-- 或：
SELECT * FROM t_account WHERE id = 1 LOCK IN SHARE MODE;  -- 也等待，X锁与S锁冲突
```

### ❌ 常见错误

- ❌ 悲观锁和乐观锁混用 → 并发控制失效

## 4. 悲观锁和乐观锁

### 🏠 生活类比

> **悲观锁** = 假设有人会抢，提前把门锁上
> **乐观锁** = 假设没人抢，操作完再检查有没有人改过

### 📖 概念讲解

| 类型 | 实现 | 适用场景 |
|------|------|---------|
| 悲观锁 | SELECT FOR UPDATE（提前加锁） | 写多冲突频繁 |
| 乐观锁 | version 字段（操作后检查） | 读多写少 |

### 💻 SQL 代码示例

```sql
USE lock_demo;

CREATE TABLE t_inventory (
    product_id INT PRIMARY KEY,
    stock INT,
    version INT DEFAULT 1
);
INSERT INTO t_inventory VALUES (1, 100, 1);

-- 悲观锁：扣库存
START TRANSACTION;
SELECT stock FROM t_inventory WHERE product_id = 1 FOR UPDATE;
-- 查到 stock=100
UPDATE t_inventory SET stock = stock - 1 WHERE product_id = 1;
COMMIT;

-- 乐观锁：更新时检查 version
START TRANSACTION;
SELECT stock, version FROM t_inventory WHERE product_id = 1;
-- 应用层检查 stock 是否足够
UPDATE t_inventory
SET stock = stock - 1, version = version + 1
WHERE product_id = 1 AND version = 1;
-- affected_rows=1 表示成功，=0 表示被其他事务更新了

-- 乐观锁检查（应用层重试逻辑）
START TRANSACTION;
SELECT stock, version FROM t_inventory WHERE product_id = 1;
-- 应用层：检查 stock 是否足够，然后更新
UPDATE t_inventory
SET stock = stock - 1, version = version + 1
WHERE product_id = 1 AND version = @old_version;
-- 检查 affected_rows，如果为0则需要重试
```

### ❌ 常见错误

- ❌ 乐观锁忘记更新 version → 并发冲突检测失效

## 5. 锁等待和死锁

### 📖 概念讲解

| 概念 | 说明 |
|------|------|
| 锁等待 | 事务 A 持有锁，事务 B 等待 A 释放锁 |
| 死锁 | 两个或多个事务相互等待对方持有的锁，形成循环 |

**InnoDB 自动检测死锁，回滚代价最小的事务**

### 💻 SQL 代码示例

```sql
USE lock_demo;

-- 死锁演示（需要两个会话）
-- Session 1:
START TRANSACTION;
UPDATE t_account SET balance = balance - 100 WHERE id = 1;  -- 锁定 id=1
-- Session 2:
START TRANSACTION;
UPDATE t_account SET balance = balance - 100 WHERE id = 2;  -- 锁定 id=2
-- Session 1:
UPDATE t_account SET balance = balance + 100 WHERE id = 2;  -- 等待 id=2 的锁
-- Session 2:
UPDATE t_account SET balance = balance + 100 WHERE id = 1;  -- 死锁！等待 id=1
-- InnoDB 检测到死锁，回滚其中一个事务

-- 查看死锁信息
SHOW ENGINE INNODB STATUS;

-- 避免死锁的方法
-- 1. 按固定顺序访问资源
-- 2. 减少事务持有锁的时间
-- 3. 使用低隔离级别
```

### ❌ 常见错误

- ❌ 业务层不做死锁重试 → 遇到死锁时直接报错

## 6. 权限管理（GRANT / REVOKE）

### 📖 概念讲解

**最小权限原则**：只授予完成任务所需的最小权限。

| 权限级别 | 说明 |
|---------|------|
| 全局 | ON *.* |
| 数据库级 | ON database_name.* |
| 表级 | ON database_name.table_name |
| 列级 | 特定列的 SELECT/INSERT/UPDATE |

### 💻 SQL 代码示例

```sql
-- -- 创建用户
-- CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'StrongPwd123!';
-- CREATE USER 'report_user'@'%' IDENTIFIED BY 'ReadOnlyPwd456!';
--
-- -- 授予全局权限
-- GRANT ALL PRIVILEGES ON *.* TO 'app_user'@'localhost';
-- GRANT SELECT ON *.* TO 'report_user'@'%';
--
-- -- 授予数据库级权限
-- GRANT SELECT, INSERT, UPDATE, DELETE ON lock_demo.* TO 'app_user'@'localhost';
--
-- -- 授予表级权限
-- GRANT SELECT, INSERT ON lock_demo.t_account TO 'app_user'@'localhost';
--
-- -- 授予列级权限
-- GRANT SELECT, UPDATE(phone, email) ON lock_demo.t_user TO 'app_user'@'localhost';
--
-- -- 撤销权限
-- REVOKE DELETE ON lock_demo.* FROM 'app_user'@'localhost';
--
-- -- 查看权限
-- SHOW GRANTS FOR 'app_user'@'localhost';
-- SHOW GRANTS FOR 'report_user'@'%';
--
-- -- 删除用户
-- DROP USER 'app_user'@'localhost';
-- DROP USER 'report_user'@'%';
```

### ❌ 常见错误

- ❌ 用 root 用户做应用连接 → 风险极大，应创建专用用户

## 7. 备份与恢复

### 📖 概念讲解

| 工具 | 类型 | 特点 |
|------|------|------|
| mysqldump | 逻辑备份 | 导出 SQL 语句，跨版本恢复 |
| mysqlpump | 逻辑备份 | mysqldump 改进版，支持并行 |
| xtrabackup | 物理备份 | 锁定少，恢复快 |

### 💻 SQL 代码示例

```sql
-- 备份单个数据库
-- mysqldump -u root -p database_name > backup.sql

-- 备份所有数据库
-- mysqldump -u root -p --all-databases > all_backup.sql

-- 备份指定表
-- mysqldump -u root -p database_name table1 table2 > tables_backup.sql

-- 恢复数据库
-- mysql -u root -p database_name < backup.sql

-- 登录 MySQL 后用 SOURCE 恢复
-- USE database_name;
-- SOURCE /path/to/backup.sql;
```

### ❌ 常见错误

- ❌ 不做备份 → 数据丢失无法恢复

## 8. binlog（二进制日志）

### 📖 概念讲解

**binlog** 记录所有修改数据的 SQL 语句，用于：
- 数据恢复（恢复到某个时间点）
- 主从复制

### 💻 SQL 代码示例

```sql
-- 查看 binlog 配置
SHOW VARIABLES LIKE 'log_bin';
SHOW VARIABLES LIKE 'binlog_format';

-- 查看 binlog 文件列表
SHOW BINARY LOGS;
SHOW MASTER STATUS;

-- 基于时间点恢复
-- mysqlbinlog binlog.000001 --stop-datetime='2024-01-15 10:00:00' | mysql -u root -p

-- 基于位置恢复
-- mysqlbinlog binlog.000001 --start-position=100 --stop-position=500 | mysql -u root -p
```

## 课后练习

**练习 1：行锁锁等待观察**
创建一个 InnoDB 表，开启两个会话：会话A：START TRANSACTION 后执行 SELECT ... FOR UPDATE 锁定某行。会话B：尝试 SELECT FOR UPDATE 同一行，观察锁等待行为。

**练习 2：乐观锁实现**
给库存表加 version 列，更新时用 UPDATE ... WHERE id=? AND version=?，用两个会话模拟并发扣减同一商品，观察 version 冲突时只有一个成功。

**练习 3：GRANT 权限操作**
创建只读用户（SELECT）和读写用户（SELECT/INSERT/UPDATE/DELETE），用 SHOW GRANTS 查看权限，然后撤销写权限。

**练习 4：死锁观察**
手动构造死锁：事务A锁X锁Y，事务B锁Y锁X，观察 InnoDB 死锁检测。

**练习 5：mysqldump 备份与恢复**
用 mysqldump 备份一个数据库，用 SOURCE 命令恢复备份。
