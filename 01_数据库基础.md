## 1. 数据库 vs 普通文件

### 🏠 生活类比

> **普通文件**就像用笔记本手写记账：你自己管理、自己解析、容易写错、没法多人同时写。
> **数据库**就像银行系统：有专业人员（DBMS）帮你管理、严格校验、支持多人同时操作、安全可靠。

### 📖 概念讲解

**数据库（Database）** 是由 DBMS（数据库管理系统）统一管理的数据集合，支持并发访问、事务安全、SQL 查询。

**普通文件**存储数据靠应用程序自己解析，无法支持并发、事务等特性。

### 💻 SQL 代码示例

```sql
-- 创建一个数据库（相当于新建一个文件夹）
CREATE DATABASE db_vs_file;
USE db_vs_file;

-- 用普通文件方式：所有数据写进一个文本文件（需要代码解析）
-- 用数据库方式：直接用 SQL 查询，无需加载整个文件

-- 创建用户表（数据库方式存储）
CREATE TABLE t_user (
    id INT PRIMARY KEY,          -- 主键：唯一标识每条记录
    name VARCHAR(50),            -- 用户名：可变字符串，最多50字符
    email VARCHAR(100)          -- 邮箱：最多100字符
);

-- 插入两条用户记录
INSERT INTO t_user VALUES (1, 'Alice', 'alice@example.com');
INSERT INTO t_user VALUES (2, 'Bob', 'bob@example.com');

-- 直接用 SQL 查询，无需解析文件
SELECT * FROM t_user WHERE id = 1;  -- 查询 id=1 的用户
```

### 📊 ASCII 对比图

```
┌─────────────────────────────────────────────────────────┐
│                    普通文件方式                          │
│  user_data.txt → 需要程序解析 → 并发冲突·数据不一致      │
├─────────────────────────────────────────────────────────┤
│                    数据库方式                            │
│  DBMS → 统一管理 → ACID保障·并发控制·SQL查询·权限管理    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      数据库                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                  │
│  │ 用户表   │  │ 订单表   │  │ 产品表   │  ← 用 SQL 查   │
│  │ t_user  │  │ t_order │  │ t_product│                  │
│  └─────────┘  └─────────┘  └─────────┘                  │
│           ↑ DBMS 统一管理 ↑                             │
│        并发控制 · 事务安全 · 权限管理                    │
└─────────────────────────────────────────────────────────┘
```

### ❌ 常见错误

- ❌ 用 Excel 文件代替数据库 → 无法支持多人同时编辑、数据一致性差
- ❌ 把数据库当文件系统用 → 滥用 TEXT 字段存 JSON，应该用正确的数据模型
- ❌ 不做备份 → 数据库崩溃数据丢失

## 2. 关系型数据库 vs 非关系型数据库

### 🏠 生活类比

> **关系型数据库**像 Excel 表格：严格的行列结构，每个格子填什么有规定。
> **非关系型数据库**像便利贴：随便贴、随便写，灵活但不好找。

### 📖 概念讲解

| 特性 | 关系型（RDBMS） | 非关系型（NoSQL） |
|------|----------------|------------------|
| 数据组织 | 表+行列+外键 | 键值/文档/列族/图 |
| 数据结构 | 高度结构化 | 灵活 Schema |
| 查询语言 | SQL 标准 | 各不相同 |
| 一致性 | ACID 强一致 | 最终一致优先 |
| 适用场景 | 银行、订单等精确数据 | 大数据、高并发 |

### 💻 SQL 代码示例

```sql
-- 关系型：结构化表，通过外键建立关系
CREATE DATABASE rdbms_vs_nosql;
USE rdbms_vs_nosql;

-- 部门表（主表）
CREATE TABLE t_department (
    dept_id INT PRIMARY KEY,    -- 主键：部门编号
    dept_name VARCHAR(50)       -- 部门名称
);

-- 员工表（子表，通过外键关联部门）
CREATE TABLE t_employee (
    emp_id INT PRIMARY KEY,    -- 主键：员工编号
    emp_name VARCHAR(50),       -- 员工姓名
    dept_id INT,               -- 外键：关联到部门表
    FOREIGN KEY (dept_id) REFERENCES t_department(dept_id)
);

-- 插入数据
INSERT INTO t_department VALUES (1, 'IT'), (2, 'HR');
INSERT INTO t_employee VALUES (1, 'Alice', 1), (2, 'Bob', 1), (3, 'Carol', 2);

-- 通过 JOIN 关联查询，体现"关系"二字
SELECT e.emp_name, d.dept_name
FROM t_employee e
INNER JOIN t_department d ON e.dept_id = d.dept_id;
```

### 📊 ASCII 表

```
关系型（RDBMS）：
┌─────────────┐       ┌─────────────┐
│ t_department │       │ t_employee  │
├─────────────┤       ├─────────────┤
│ dept_id (PK)│◄──────│ dept_id (FK)│
│ dept_name   │       │ emp_id (PK) │
└─────────────┘       │ emp_name    │
                       └─────────────┘
  每个员工 → 属于一个部门 → 数据结构清晰

非关系型（NoSQL）—— 以文档型为例：
{
  "_id": "emp_001",
  "name": "Alice",
  "dept": { "id": 1, "name": "IT" }  -- 直接嵌入，无需外键
}
```

### ❌ 常见错误

- ❌ 所有场景都用 NoSQL → 金融、订单等强一致场景必须用关系型
- ❌ 所有场景都用 RDBMS → 日志、缓存等灵活 Schema 场景用 NoSQL 更合适

## 3. SQL 到底是什么

### 🏠 生活类比

> SQL 就像点餐时的语言：你说"我要一份宫保鸡丁加米饭"，服务员就给你端来。**你不需要告诉厨师怎么炒菜**，只需要说"要什么"。

### 📖 概念讲解

**SQL（Structured Query Language）** 是用于操作关系型数据库的标准编程语言，是一种**声明式语言**。

- **声明式** = 你告诉数据库"要什么"（What），不需要写"怎么找"（How）
- 相比之下，Java/Python 是命令式语言，需要一步一步告诉计算机怎么做

### 💻 SQL 代码示例

```sql
CREATE DATABASE sql_demo;
USE sql_demo;

-- 创建产品表
CREATE TABLE t_products (
    id INT PRIMARY KEY AUTO_INCREMENT,  -- 主键自增
    product_name VARCHAR(100),            -- 产品名称
    price DECIMAL(10,2),                  -- 价格（10位数字，2位小数）
    stock INT,                           -- 库存
    category VARCHAR(50)                   -- 分类
);

-- 插入多条产品记录
INSERT INTO t_products (product_name, price, stock, category) VALUES
    ('iPhone', 6999.00, 100, 'Electronics'),
    ('MacBook Pro', 12999.00, 50, 'Electronics'),
    ('Desk Chair', 599.00, 200, 'Furniture'),
    ('Standing Desk', 1299.00, 80, 'Furniture'),
    ('Wireless Mouse', 129.00, 500, 'Electronics');

-- 查询：SELECT ... FROM ... WHERE ...
-- 查询电子产品中价格小于5000的
SELECT product_name, price
FROM t_products
WHERE category = 'Electronics' AND price < 5000;

-- 统计：按分类聚合
SELECT
    category,                          -- 分类
    COUNT(*) AS product_count,         -- 产品数量
    AVG(price) AS avg_price           -- 平均价格
FROM t_products
GROUP BY category;                     -- 按分类分组
```

### ❌ 常见错误

- ❌ 把 SQL 当作编程语言写复杂算法 → SQL 擅长数据操作，不是通用编程语言
- ❌ 在 SQL 里嵌套太多层子查询 → 应该用 WITH CTE 或拆分成多步

## 4. 主键与外键

### 🏠 生活类比

> **主键**就像人的身份证号 —— 唯一标识每一个人，不能重复，不能为空。
> **外键**就像学生证上的学校编号 —— 引用学校的存在，确保学生属于某个学校。

### 📖 概念讲解

| 项目 | 主键（Primary Key） | 外键（Foreign Key） |
|------|---------------------|---------------------|
| 数量 | 每表只能一个 | 每表可以多个 |
| 唯一性 | 必须唯一 | 可以重复 |
| NULL | 不允许 NULL | 允许 NULL（独立存在） |
| 作用 | 唯一标识记录 | 建立表间关联 |

### 💻 SQL 代码示例

```sql
CREATE DATABASE pk_fk_demo;
USE pk_fk_demo;

-- 班级表：dept_id 是主键（班级的唯一标识）
CREATE TABLE t_department (
    dept_id INT PRIMARY KEY,     -- 主键：班级编号
    dept_name VARCHAR(50) NOT NULL  -- 班级名称，非空
);

-- 学生表：student_id 是主键，dept_id 是外键
CREATE TABLE t_student (
    student_id INT PRIMARY KEY,   -- 主键：学生编号
    student_name VARCHAR(50),    -- 学生姓名
    dept_id INT,                 -- 外键：所属班级
    FOREIGN KEY (dept_id) REFERENCES t_department(dept_id)  -- 外键约束
);

-- 插入数据
INSERT INTO t_department VALUES (1, 'Computer Science'), (2, 'Mathematics');
INSERT INTO t_student VALUES (1, 'Alice', 1), (2, 'Bob', 1), (3, 'Carol', 2);

-- 外键作用：dept_id 只能引用存在的班级编号
-- 下面这条会报错，因为 dept_id = 99 不存在
-- INSERT INTO t_student VALUES (4, 'David', 99);  -- Error!
```

### 📊 ASCII 图

```
t_department（班级表）          t_student（学生表）
┌───────────────┐              ┌───────────────────────┐
│ dept_id (PK)  │◄────────────│ dept_id (FK)          │
│ dept_name     │   外键引用    │ student_id (PK)       │
└───────────────┘              │ student_name          │
   dept_id=1: CS                 └───────────────────────┘
   dept_id=2: Math

Alice(student_id=1) → dept_id=1 → CS 班
Carol(student_id=3) → dept_id=2 → Math 班
```

### ❌ 常见错误

- ❌ 用业务字段（如邮箱、用户名）做主键 → 业务可能变，建议用自增 ID
- ❌ 外键不建索引 → 大数据量时关联查询性能差
- ❌ 外键引用不存在的父记录 → 数据库报错，应该先检查父表

## 5. 数据库设计三大范式

### 🏠 生活类比

> 三大范式就像整理房间的规则：
> **1NF**：每个抽屉只放一类东西（不能混放）
> **2NF**：一个抽屉只放一个主人的东西（不共享）
> **3NF**：抽屉标签只描述抽屉本身（不传递）

### 📖 概念讲解

| 范式 | 核心要求 | 解决的问题 |
|------|----------|-----------|
| **1NF** | 每个字段原子值，不可再分 | 地址字段拆成省-市-区 |
| **2NF** | 非主键字段完全依赖主键 | 联合主键下的部分依赖 |
| **3NF** | 非主键字段只依赖主键 | 传递依赖（A→B→C） |

### 💻 SQL 代码示例

```sql
CREATE DATABASE normalization_demo;
USE normalization_demo;

-- 满足1NF：字段都是原子值
CREATE TABLE t_customer (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(50),
    province VARCHAR(50),   -- 省（原子值）
    city VARCHAR(50),       -- 市（原子值）
    district VARCHAR(50)    -- 区（原子值）
);
INSERT INTO t_customer VALUES (1, 'Alice', 'Guangdong', 'Shenzhen', 'Nanshan');

-- 满足2NF：拆分为课程表和选课表（解决部分依赖）
CREATE TABLE t_course (
    course_id INT PRIMARY KEY,    -- 主键
    course_name VARCHAR(50),
    teacher_name VARCHAR(50)      -- 只依赖 course_id，不部分依赖
);
CREATE TABLE t_student_course (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id)  -- 联合主键
);

-- 满足3NF：班级信息独立成表（解决传递依赖）
CREATE TABLE t_class (
    class_id INT PRIMARY KEY,
    class_name VARCHAR(50)       -- 只依赖 class_id
);
CREATE TABLE t_student2 (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(50),
    class_id INT                 -- 只依赖主键 student_id
    -- class_name 不在这里（会传递依赖），应该放在 t_class
);
```

### ❌ 常见错误

- ❌ 过度追求范式 → 3NF 以上可能导致大量 JOIN，影响查询性能
- ❌ 不做反范式 → 大数据量报表场景需要冗余存储减少 JOIN

## 6. 反范式设计

### 🏠 生活类比

> 买东西有两种方式：
> **范式化** = 去超市买原料回家做（每次都要去多个地方买原料）
> **反范式** = 直接买预制菜（贵一点但方便，数据冗余但查询快）

### 📖 概念讲解

**反范式**是有意违反范式，通过冗余数据提升查询性能。

| 对比 | 范式化设计 | 反范式设计 |
|------|-----------|-----------|
| 数据冗余 | 无冗余 | 有冗余 |
| 更新操作 | 只需改一处 | 需要改多处 |
| 查询操作 | 可能需要 JOIN | 直接查，少量 JOIN |
| 适用场景 | 写入频繁 | 读取频繁 |

### 💻 SQL 代码示例

```sql
USE normalization_demo;

-- 范式化设计：订单表只存 user_id，用户信息在用户表
CREATE TABLE t_user_norm (
    user_id INT PRIMARY KEY,
    user_name VARCHAR(50)
);
CREATE TABLE t_order_norm (
    order_id INT PRIMARY KEY,
    user_id INT,
    order_amount DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES t_user_norm(user_id)
);

-- 反范式化：订单表冗余存储 user_name，无需 JOIN
CREATE TABLE t_order_denorm (
    order_id INT PRIMARY KEY,
    user_id INT,
    user_name VARCHAR(50),       -- 冗余：避免 JOIN
    order_amount DECIMAL(10,2)
);

INSERT INTO t_user_norm VALUES (1, 'Alice');
INSERT INTO t_order_norm VALUES (1, 1, 299.00);
INSERT INTO t_order_denorm VALUES (1, 1, 'Alice', 299.00);

-- 范式查询（需 JOIN）
SELECT u.user_name, o.order_amount
FROM t_user_norm u
INNER JOIN t_order_norm o ON u.user_id = o.user_id;

-- 反范式查询（无需 JOIN）
SELECT user_name, order_amount FROM t_order_denorm;
```

### ❌ 常见错误

- ❌ 所有表都做反范式 → 数据更新时代价太大，要同步多处
- ❌ 反范式后不做一致性维护 → 冗余字段和原始字段值不一致

## 7. 常见关系型数据库与 MySQL 特点

### 📖 概念讲解

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| **MySQL** | 开源免费，轻量，Web 首选 | Web 应用 |
| **PostgreSQL** | 功能最丰富，学术/企业级 | 企业级/分析 |
| **Oracle** | 商业，功能最全，昂贵 | 金融/大型企业 |
| **SQLite** | 嵌入式，零配置 | 嵌入式/移动/测试 |

### 💻 SQL 代码示例

```sql
-- 查看 MySQL 支持的存储引擎
SHOW ENGINES;

-- MySQL 默认使用 InnoDB 引擎，具备以下特性：
-- 1. 支持事务（ACID）
-- 2. 支持行级锁
-- 3. 支持外键
-- 4. 支持 MVCC（多版本并发控制）
-- 5. 崩溃自动恢复
```

### 📊 MySQL vs 其他数据库语法速查

| 特性 | MySQL | PostgreSQL | Oracle | SQLite |
|------|-------|-----------|--------|--------|
| 自增 ID | AUTO_INCREMENT | SERIAL | SEQUENCE | AUTOINCREMENT |
| 分页 | LIMIT | LIMIT/OFFSET | FETCH FIRST | LIMIT |
| 空值函数 | IFNULL | COALESCE | NVL | IFNULL |
| Upsert | ON DUPLICATE KEY | ON CONFLICT | MERGE | INSERT OR REPLACE |
| 批量插入 | VALUES(),(),() | 同 MySQL | 同 MySQL | 同 MySQL |
