## 1. ACID 四大特性

### 🏠 生活类比

> **原子性** = 转账要么全成功，要么全失败
> **一致性** = 转账前后余额总和不变
> **隔离性** = 两人同时转账互不干扰
> **持久性** = 转账成功后记录永久保存

### 📖 概念讲解

| 特性 | 说明 |
|------|------|
| Atomicity（原子性） | 事务是最小执行单元，要么全部成功，要么全部失败 |
| Consistency（一致性） | 事务前后数据库状态保持一致，满足所有约束 |
| Isolation（隔离性） | 并发事务之间相互隔离，互不干扰 |
| Durability（持久性） | 提交后的修改永久保存，即使系统崩溃也不丢失 |

### 💻 SQL 代码示例

```sql
CREATE DATABASE transaction_demo;
USE transaction_demo;

CREATE TABLE t_account (
    account_id INT PRIMARY KEY,
    account_name VARCHAR(50),
    balance DECIMAL(10,2)
);
INSERT INTO t_account VALUES (1, 'Alice', 10000), (2, 'Bob', 5000);

-- 转账场景演示原子性和一致性
START TRANSACTION;
UPDATE t_account SET balance = balance - 3000 WHERE account_id = 1;
UPDATE t_account SET balance = balance + 3000 WHERE account_id = 2;
COMMIT;

-- 验证余额总和不变（一致性）
SELECT SUM(balance) AS total FROM t_account;  -- 15000

-- 测试原子性
START TRANSACTION;
UPDATE t_account SET balance = balance - 5000 WHERE account_id = 1;
-- 模拟错误
ROLLBACK;
SELECT * FROM t_account;  -- Alice 余额未变
```

### ❌ 常见错误

- ❌ 忘记 COMMIT → 断开连接后数据丢失

## 2. 并发环境下的问题

### 📖 概念讲解

| 问题 | 说明 | 读到什么 |
|------|------|--------|
| 脏读 | 读取到其他事务未提交的数据 | 未提交的修改 |
| 不可重复读 | 同一事务中两次读取同一行，数据不一致 | 已提交的新值 |
| 幻读 | 同一事务中两次查询，结果集行数不同 | 已提交的新插入行 |

### 📊 ASCII 图

```
脏读：Session1 修改但未提交，Session2 读到了
  Session1: UPDATE balance = 9000（未提交）
  Session2: SELECT balance → 9000（脏读）

不可重复读：Session1 两次读取，Session2 修改并提交了
  Session1: SELECT balance → 10000
  Session2: UPDATE balance = 5000 → COMMIT
  Session1: SELECT balance → 5000（同一事务中两次结果不同）

幻读：Session1 两次查询，Session2 插入了新行
  Session1: SELECT COUNT(*) FROM t → 5
  Session2: INSERT ... → COMMIT
  Session1: SELECT COUNT(*) FROM t → 6（同一事务中行数多了）
```

## 3. 四大隔离级别

### 📖 概念讲解

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| READ UNCOMMITTED | ✅ 可能 | ✅ 可能 | ✅ 可能 |
| READ COMMITTED | ❌ 不可能 | ✅ 可能 | ✅ 可能 |
| REPEATABLE READ（MySQL默认） | ❌ 不可能 | ❌ 不可能 | ✅ 可能（InnoDB防幻读） |
| SERIALIZABLE | ❌ 不可能 | ❌ 不可能 | ❌ 不可能 |

### 💻 SQL 代码示例

```sql
USE transaction_demo;

-- 查看当前隔离级别
SELECT @@transaction_isolation;
SELECT @@global.transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- InnoDB 默认 REPEATABLE READ，通过 MVCC 防止幻读
CREATE TABLE t_counter (id INT PRIMARY KEY, value INT);
INSERT INTO t_counter VALUES (1, 0);
```

### ❌ 常见错误

- ❌ 使用最低隔离级别 → 可能读到脏数据

## 4. START TRANSACTION / COMMIT / ROLLBACK / SAVEPOINT

### 💻 SQL 代码示例

```sql
USE transaction_demo;

-- 基础事务操作
START TRANSACTION;
INSERT INTO t_account VALUES (3, 'Carol', 3000);
COMMIT;  -- 持久化

START TRANSACTION;
INSERT INTO t_account VALUES (4, 'David', 4000);
ROLLBACK;  -- 撤销

SELECT * FROM t_account;  -- 只有 Alice、Bob、Carol

-- SAVEPOINT 演示
START TRANSACTION;
INSERT INTO t_account VALUES (5, 'Eve', 5000);
SAVEPOINT sp1;
INSERT INTO t_account VALUES (6, 'Frank', 6000);
SAVEPOINT sp2;
INSERT INTO t_account VALUES (7, 'Grace', 7000);

-- 回滚到 sp2
ROLLBACK TO SAVEPOINT sp2;
-- Grace 被撤销，Eve 和 Frank 还在

-- 再回滚到 sp1
ROLLBACK TO SAVEPOINT sp1;
-- Frank 也被撤销，只剩 Eve

COMMIT;
SELECT * FROM t_account;  -- 5条记录
```

### ❌ 常见错误

- ❌ 嵌套事务 → MySQL 不支持，应该用 SAVEPOINT

## 5. MVCC（多版本并发控制）

### 📖 概念讲解

MVCC = Multi-Version Concurrency Control

**原理**：
- 每行数据有两个隐藏列：DB_TRX_ID（最后修改事务ID）、DB_ROLL_PTR（回滚指针）
- 每次修改时，数据行复制到 undo log，回滚指针指向旧版本
- 读操作根据事务开始的快照读取对应版本的数据
- 使得**读不被写阻塞，写不被读阻塞**

### 📊 ASCII 图

```
┌─────────────────────────────────────────────────┐
│                  MVCC 原理                       │
├─────────────────────────────────────────────────┤
│ 数据行：Alice balance=10000                     │
│                                                  │
│ 事务A 修改：Alice balance=9000                  │
│   旧版本: Alice balance=10000 → undo log       │
│   新版本: Alice balance=9000 → 数据行          │
│                                                  │
│ 事务B 读取（REPEATABLE READ）：                 │
│   → 读取 undo log 中的旧版本 → 10000          │
│                                                  │
│ 事务C 读取（READ COMMITTED）：                  │
│   → 读取已提交的新版本 → 9000                 │
└─────────────────────────────────────────────────┘
```

## 6. Redo Log 和持久性

### 📖 概念讲解

**Redo Log（重做日志）**：
- 记录已提交事务的修改，用于崩溃恢复
- 写入流程：修改数据 → 写入 redo log → COMMIT → 数据落盘
- 即使数据库崩溃，也可通过 redo log 恢复

### 💻 SQL 代码示例

```sql
USE transaction_demo;

-- 查看 redo log 配置
SHOW VARIABLES LIKE 'innodb_log%';
```

### ❌ 常见错误

- ❌ 认为关闭数据库不会丢数据 → 如果没有提交且 redo log 未刷盘，数据丢失

## 课后练习

**练习 1：事务与 SAVEPOINT**
开启事务：向账户表插入一条记录，然后设置 SAVEPOINT，再插入第二条记录并回滚到 SAVEPOINT，最后 COMMIT。验证哪些记录被持久化。

**练习 2：并发锁观察**
模拟并发场景（需要两个会话）：会话A开启事务修改某行数据不提交，会话B尝试修改同一行，观察会话B是被阻塞还是立即报错。

**练习 3：隔离级别对比**
查询当前会话和全局的隔离级别，然后分别设置 READ COMMITTED 和 SERIALIZABLE，用 BEGIN 开启事务后查询数据，观察两者行为差异。

**练习 4：部分回滚**
用 ROLLBACK TO SAVEPOINT 演示部分回滚：在一个事务中依次插入3条记录，分别设置 sp1 和 sp2 两个保存点，回滚到 sp1 后 COMMIT，验证哪几条记录被保存。

**练习 5：转账一致性验证**
用 START TRANSACTION + COMMIT + ROLLBACK 实现一个完整的转账事务：账户A减100，账户B加100，验证转账前后两账户余额总和不变（一致性验证）。
